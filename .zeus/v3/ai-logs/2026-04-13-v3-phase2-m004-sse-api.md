# v3 Phase 2 M-004 — Real-time Event Bus + FastAPI Server

## Decision Rationale

Phase 1 gave v3 a runnable CLI and robust worker pipeline. Phase 2's first milestone (M-004) adds the real-time layer that makes v3 observable and integratable with external frontends and CI systems. The key deliverables were:
1. An **in-memory event bus** that can broadcast execution events to multiple subscribers
2. A **FastAPI server** exposing REST + SSE endpoints
3. **Metrics summary** endpoint (M-005 groundwork) built on the existing `EventLog` table
4. **Bidirectional wiring**: scheduler and workers now emit events to the bus in addition to persisting them to the database

## Execution Summary

### New files
- `.zeus/v3/scripts/api/bus.py` — `EventBus`
  - Maintains a set of `asyncio.Queue` subscribers
  - `emit()` broadcasts fire-and-forget messages
  - `subscribe()` yields SSE-formatted `data: {...}\n\n` chunks
- `.zeus/v3/scripts/api/server.py` — FastAPI app factory `create_app(store, bus)`
  - `GET /health` — store health metadata
  - `GET /tasks` & `GET /tasks/{task_id}` — task listing and detail
  - `GET /events` — paginated event query backed by `EventLog` table
  - `GET /events/stream` — `StreamingResponse` with `text/event-stream`
  - `GET /metrics/summary` — task counts, quarantine count, pass rate
- `.zeus/v3/scripts/api/__init__.py` — public exports
- `.zeus/v3/scripts/tests/test_v3_api.py` — API + SSE + EventBus tests

### Modified files
- `.zeus/v3/scripts/core/scheduler.py` — `ZeusScheduler` now accepts optional `bus` argument and emits `task.enqueued` + `global.completed`
- `.zeus/v3/scripts/core/worker.py` — `ZeusWorker` accepts optional `bus` and emits `task.started`, `task.completed`, `task.failed`
- `.zeus/v3/scripts/core/worker_pool.py` — `WorkerPool` forwards `bus` to each worker
- `.zeus/v3/scripts/run.py` — new `--serve` / `--host` / `--port` flags:
  - `python run.py --serve` starts the API server
  - Runner mode automatically creates an `EventBus` and wires it into scheduler + pool

### Design notes
- The `EventBus` is intentionally single-process. For multi-node deployments it can be swapped for a Redis Pub/Sub implementation without changing the scheduler/worker interface.
- `store.log_event()` remains the durable source of truth; `bus.emit()` is the ephemeral real-time projection. This gives us both durability and low-latency broadcasting.
- `TestClient` cannot be used for `StreamingResponse` inside an async pytest because it blocks the event loop. We validated the SSE endpoint by checking route registration and tested `EventBus.subscribe()` directly.
