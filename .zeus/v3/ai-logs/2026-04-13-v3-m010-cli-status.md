# v3 M-010 Pre-work — zeus_runner.py v3 Status Integration

## Decision Rationale

M-010 calls for upgrading `zeus_runner.py` to serve as a compatibility frontend for v3. The most immediate pain point is that `zeus_runner.py --status` only reads v1/v2 `task.json` files and is completely blind to v3's SQLite database state. Making the runner v3-aware allows operators to use the same familiar CLI entrypoint across all project versions.

## Execution Summary

### New files
- `.zeus/v3/config.json` — default v3 configuration so that `ZeusConfig` and `zeus_runner.py` have a readable config source

### Modified files
- `.zeus/scripts/zeus_runner.py` — v3 status support
  - Added `.zeus/v3/scripts` to `sys.path` so v3 modules can be imported
  - `status()` now branches to `_status_v3()` when `--version v3` is passed
  - `_status_v3()`:
    - Imports `SQLiteStateStore` dynamically
    - Auto-discovers the v3 database file (`state.db` or fallback `zeus_open_v3.sqlite`)
    - Reads task states and quarantine count asynchronously
    - Prints `completed/failed/pending/running` breakdown
    - Lists recent completed and next pending (or currently running) tasks
    - Recommends the v3 CLI (`run.py --serve`) when tasks are in flight

### Verification
- `python .zeus/scripts/zeus_runner.py --status` (main) → v2 status output unchanged ✅
- `python .zeus/scripts/zeus_runner.py --status --version v3` → v3 database status output ✅

## Target Impact

- **developer_adoption_rate** ↑↑↑ — Operators can use the same `zeus_runner.py` for both legacy and v3 projects
- **observability** ↑↑ — v3 state is now visible from the command line without starting the API server
