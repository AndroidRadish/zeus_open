# ZeusOpen Agent 设定改进方案

> 对比对象：`trueproject/zeus-open/AGENTS.md` + `.zeus/v3/agent-workspaces/*/PROMPT.md`
> 改进依据：Andrej Karpathy 的 LLM 编码行为四大原则（Think Before Coding / Simplicity First / Surgical Changes / Goal-Driven Execution）

---

## 一、现有 Agent 设定的结构梳理

ZeusOpen v3 的 Agent 体系由两层文档构成：

### 1.1 主层：`zeus-open/AGENTS.md`
作用于所有与 zeus-open 交互的 Agent（包括主会话 Agent 和子 Agent 的初始化认知）。

**现有章节：**
- 开工流程（pwd → 读 progress → 读 task.json → git log → zeus_runner --status）
- 工作规则（一次一个 task、防过度延伸、依赖仓库文件而非聊天记录）
- Zeus 工作流（init → discover → brainstorm → plan → execute → feedback → evolve）
- 常用命令（status / plan / 执行 wave / 执行 task）
- 完成定义（7 项检查清单：功能实现、测试通过、lint、task.json 更新、AI log、git commit、runner 状态正常）
- Agent 映射（6 种专业 agent 与 Kimi Agent 类型的对应）
- 收尾流程（更新 progress / task.json / 审查 / commit / status 检查）
- 问题排查（runner 报错、遇到 blocker 的处理）

### 1.2 任务层：`.zeus/v3/agent-workspaces/*/PROMPT.md`
作用于具体的子 Agent 实例（如 zeus-agent-T-V3-001），每次派生子 Agent 时动态注入。

**现有内容：**
- 项目信息（名称、北极星指标、版本）
- 当前任务（ID、Wave、类型、标题、描述、涉及文件、依赖任务）
- 执行要求（隔离工作区、遵循现有风格、typecheck/lint/test、写 ai-log、输出 zeus-result.json、不 git commit、不修改主 task.json）
- 进度上报（A 方案 progress.jsonl / B 方案 HTTP POST）

---

## 二、现有优势（应保留）

| 维度 | 现有体现 | 评价 |
|------|---------|------|
| **流程仪式感强** | 开工 6 步、收尾 5 步、完成定义 7 项 | 良好，确保 Agent 不盲动 |
| **任务粒度控制** | "一次只做一个功能"、"防过度延伸"四步法 | 良好，防止范围蔓延 |
| **结果可追踪** | 要求写 ai-log、zeus-result.json、progress.jsonl | 优秀，解决黑盒问题 |
| **Agent 专业化映射** | researcher/planner/executor 等 6 种角色 | 优秀，便于调度器分派 |

## 三、核心盲区（亟需改进）

### 盲区 1：认知启动不足 — 缺少「Think Before Coding」

**现状：**
- AGENTS.md 的开工流程要求读文件、查状态，但没有要求 Agent 在**理解任务本身**上做任何显式校准
- PROMPT.md 直接给出任务描述和执行要求，没有要求子 Agent 列出假设或提问

**后果：**
- 子 Agent 拿到任务描述后，如果某个接口用法、文件路径、业务逻辑存在歧义，会直接按猜测执行
- 等返回 zeus-result.json 后，主 Agent 才发现理解偏差，造成大量返工

**案例：**
T-V3-019（hot reload）涉及 `server.py`、`control_plane.py`、`importer.py` 等多个文件。如果子 Agent 对"Protect against concurrent scheduler ticks during reload"的理解有偏差，可能直接实现一个复杂的锁机制，而实际项目可能已经有现成的状态机可用。

---

### 盲区 2：代码复杂度无约束 — 缺少「Simplicity First」

**现状：**
- AGENTS.md 和 PROMPT.md 都没有对"代码应该写多长"、"是否应该引入抽象"做任何约束
- 仅有"遵循项目现有的代码风格和架构模式"，这反而可能被 LLM 误解为"如果项目里有策略模式，我也可以继续用策略模式"

**后果：**
- 子 Agent 特别容易在隔离工作区中写出过度工程的代码
- 50 行能解决的问题写成 500 行，合并回主仓库时增加 review 负担和长期维护成本

**案例：**
子 Agent 在实现 T-V3-001（进度上报）时，可能会为"文件扫描进度"和"HTTP 推送进度"分别设计两个抽象基类 `ProgressReporter` 和 `ProgressForwarder`，并配上一套 `ProgressConfig` dataclass。而实际上，直接在两处各写 10 行逻辑就够了。

---

### 盲区 3：修改范围无边界 — 缺少「Surgical Changes」

**现状：**
- AGENTS.md 有"一次只做一个功能"，但这约束的是**功能范围**，不是**代码 diff 范围**
- 没有禁止"顺手改进相邻代码、注释、格式、类型提示"

**后果：**
- 在多子 Agent 并行工作的 v3 架构下，Agent A 顺手改了某文件的风格，Agent B 也在改同一个文件的业务逻辑，合并时极易产生无意义的冲突
- 主 Agent review 时需要花费大量时间区分"任务相关改动"和"顺手改动"

**案例：**
子 Agent 修复 `api/server.py` 中的某个 bug 时，顺手把文件中其他函数的 `print` 改成了 `logger.info`，调整了引号风格，加了类型提示。这些改动与任务无关，但会出现在 diff 中。

---

### 盲区 4：验证哲学薄弱 — 缺少「Goal-Driven Execution」

**现状：**
- 完成定义中有"测试全部通过"，但这是**结果检查**
- 没有要求把"修复 bug"转化为"先写复现测试，再修复"，也没有要求多步骤任务按"步骤 → 验证点"格式列出计划

**后果：**
- 子 Agent 倾向于"先改代码，再想办法让测试通过"，甚至绕过测试
- 对于跨多个文件的任务（如 T-V3-019 涉及 7 个文件），子 Agent 可能一次性全部改完后再统一测试，中间出错难以定位

**案例：**
子 Agent 接到"修复空邮箱验证"的任务，直接修改了验证逻辑，没有先写一个输入为空字符串的测试。结果修改虽然让原有测试通过了，但没有覆盖 `.strip()` 的边界情况，主仓库合并后仍可能出问题。

## 四、分 layer 改进方案

### 4.1 AGENTS.md 改进（主 Agent + 子 Agent 通用认知）

#### 4.1.1 新增：🧠 动手前四问（Think Before Coding）

**插入位置：**开工流程之前

```markdown
## 🧠 动手前四问（Think Before Coding）

在编写或修改任何代码之前，必须显式回答以下问题。
如果答案是"不确定"或"有歧义"，**必须先提问，不要直接实现**。

1. **我做了什么假设？**
   - 用户需求中哪些信息是我推测的而非明确给出的？
   - 如果推测可能出错，我会先提问。

2. **是否存在多种理解方式？**
   - 如果指令有歧义，列出所有可能的解释，让用户（或调度器）选择。
   - **禁止在存在歧义时默默选一个然后开始做。**

3. **有没有更简单的做法？**
   - 如果我发现一个更简单的方案能达到同样效果，必须主动提出。
   - 不要为了实现"更好的架构"而增加不必要的复杂度。

4. **我对哪里感到困惑？**
   - 如果某个依赖、接口或业务逻辑不清楚，必须停下来提问。
   - 禁止带着困惑蒙混过关。
```

#### 4.1.2 开工流程插入第 0 步

将现有开工流程：
```
1. pwd
2. 读 progress
3. 读 task.json
...
```

改为：
```markdown
## 🚀 开工流程

**写代码前必须按顺序执行：**

0. 回答上面的「动手前四问」（如有歧义/困惑，先向用户/调度器提问）
1. `pwd` — 确认当前在项目根目录
2. 读取 `claude-progress.md`（或 `.zeus/main/evolution.md`）— 了解最新状态
3. 读取 `.zeus/main/task.json` — 查看待完成任务
4. `git log --oneline -5` — 查看最近提交
5. 运行状态检查：`python .zeus/scripts/zeus_runner.py --status`
6. 如果基础验证失败，先修复基础状态
```

#### 4.1.3 新增：✂️ 代码简洁性三原则（Simplicity First）

**插入位置：**「工作规则」之后、「Zeus 工作流」之前

```markdown
## ✂️ 代码简洁性三原则（Simplicity First）

**默认规则：用最小代码解决当前问题。不为未来做 speculation。**

### 原则 1：不超需求加功能
- 只实现明确要求的内容，以及为了该功能"能跑通"所必需的最小配套代码。
- 不要提前实现"以后可能用得上的"配置项、开关、通用接口。

### 原则 2：不为一行代码建抽象
- 如果一个抽象（类、接口、策略模式、工厂）只为当前这一个用例服务，不要建。
- 直接写函数。当同一个逻辑真的出现了第 3 次重复时，再考虑提取。

### 原则 3：不写不可能触发的错误处理
- 不要为"理论上可能但实际永远不会发生"的情况写复杂的防御代码。
- 如果无法判断某个异常是否真会发生，先实现最简单的路径，并在注释中标记：`# TODO: 评估此处是否需要处理 X 异常`。

### 自检问题
在提交任何代码前，问自己：
> "如果一位资深工程师 review 这段代码，他会不会说'这太过度设计了'？"
> 如果答案是"会"，立即简化。
```

#### 4.1.4 新增：🔪 手术式修改规范（Surgical Changes）

**插入位置：**「代码简洁性三原则」之后

```markdown
## 🔪 手术式修改规范（Surgical Changes）

**核心原则：只碰必须碰的。只清理自己制造的。**

### 规则 1：禁止顺手改进相邻代码
当任务要求"修复 A"时：
- ❌ 不要重构与 A 无关的函数 B
- ❌ 不要为相邻的类添加类型提示
- ❌ 不要修改无关的注释和文档字符串
- ❌ 不要调整引号风格、缩进或换行

### 规则 2：匹配现有风格
即使你认为另一种风格更好，也要匹配当前文件的：
- 引号风格（`'` vs `"`）
- 类型提示使用习惯（有或无）
- 异常处理方式
- 命名惯例

### 规则 3：orphan 清理
- 如果你的修改导致某些 import / 变量 / 函数不再被使用 → **必须删除**
- 预先存在的死代码 / 未使用 import → **不要动**（除非 task 明确要求清理）

### 规则 4：diff 自检
在标记任务完成之前，必须在脑海中检查：
> "每一行改动是否都能直接追溯到用户的这个请求？"
> 如果有任何一行不能，撤销它。
```

#### 4.1.5 新增：🎯 验证循环工作法（Goal-Driven Execution）

**插入位置：**「手术式修改规范」之后、「Zeus 工作流」之前

```markdown
## 🎯 验证循环工作法（Goal-Driven Execution）

**核心原则：把指令转化为可验证的目标，循环直到通过。**

### 指令转换表

收到以下类型的请求时，必须按右侧方式执行：

| 收到的指令 | 必须转化成的目标 |
|-----------|-----------------|
| "修复 bug" | 1) 写一个能复现该 bug 的测试<br>2) 运行测试，确认它失败<br>3) 修复代码<br>4) 运行测试，确认它通过 |
| "添加功能" | 1) 明确该功能的输入和预期输出<br>2) 先写测试（或最小可运行脚本）<br>3) 实现功能让测试通过 |
| "重构 X" | 1) 运行现有测试，记录通过数<br>2) 执行重构<br>3) 运行测试，确保通过数不变或增加 |
| "优化性能" | 1) 先写基准测试或测量当前耗时<br>2) 实施优化<br>3) 再次测量，确认有提升 |

### 多步骤任务的计划格式

对于涉及多个文件的复杂任务，在动手前必须给出简要计划：

```
1. [步骤描述] → 验证：[如何确认这一步成功]
2. [步骤描述] → 验证：[如何确认这一步成功]
3. [步骤描述] → 验证：[如何确认这一步成功]
```

**示例（Dashboard 功能补全）：**
```
1. 分析后端 API 对 MetricsPanel 的数据需求 → 验证：列出所有需要调用的端点
2. 实现 MetricsPanel Vue 组件 → 验证：组件能正确渲染 mock 数据
3. 对接后端 API → 验证：真实数据能正常加载，无 TypeScript 报错
4. 运行测试 → 验证：pytest 和前端 build 均通过
```

**禁止行为：**
- ❌ "我先看看代码，然后改改看"
- ❌ 一次性修改 5 个文件后再统一测试
- ❌ 用"应该可以了"代替实际验证
```

#### 4.1.6 完成定义的改进

**现有完成定义：**
```markdown
- [ ] 目标行为已经实现
- [ ] 测试全部通过
- [ ] Lint / 类型检查无错误
- [ ] `task.json` 已更新（`passes: true`, `commit_sha` 已填）
- [ ] AI log 已写入 `.zeus/main/ai-logs/`
- [ ] 代码已 `git commit`（格式：`feat(T-001): description`）
- [ ] `zeus_runner.py --status` 显示正常
```

**改进后：**
```markdown
## ✅ 完成定义

一个 task 只有在以下条件**全部满足**时才算完成：

- [ ] 目标行为已经实现
- [ ] **修复类 task**：已先写复现测试，测试从"失败"变"通过"
- [ ] **功能类 task**：已写对应测试/可运行脚本，且通过
- [ ] 测试全部通过
- [ ] Lint / 类型检查无错误
- [ ] `task.json` 已更新（`passes: true`, `commit_sha` 已填）
- [ ] AI log 已写入 `.zeus/main/ai-logs/`
- [ ] 代码已 `git commit`（格式：`feat(T-001): description`）
- [ ] `zeus_runner.py --status` 显示正常
- [ ] **diff 自检通过**：每一行改动都能直接追溯到当前 task 需求
```

### 4.2 PROMPT.md 改进（子 Agent 任务特定提示）

PROMPT.md 是子 Agent 的"军令状"，需要把 AGENTS.md 中的高质量编码原则压缩成几条不可违背的铁律，直接注入到每次子 Agent 会话中。

#### 4.2.1 现有 PROMPT.md 的问题

以 T-V3-001 的 PROMPT.md 为例：
```markdown
## 执行要求
1. 在隔离工作区中实现上述任务，遵循项目现有的代码风格和架构模式。
2. 如有 typecheck / lint / test 配置，运行并保证通过。
3. 在 `.zeus/v3/ai-logs/` 目录写入 ai-log 文件...
4. 完成后必须写入 `zeus-result.json`...
5. 不需要在隔离工作区中执行 `git commit`...
```

问题：
- 第 1 条的"遵循现有架构模式"太容易被解读为"可以继续过度设计"
- 没有要求子 Agent 在动手前显式列出假设
- 没有要求测试先行
- 没有约束 diff 范围

#### 4.2.2 改进后的 PROMPT.md 模板

以下是可直接套用到所有 v3 子 Agent PROMPT.md 中的改进版「执行要求」：

```markdown
## 执行要求

### 0. 认知校准（Think Before Coding）
在动手写代码前，你必须：
- 阅读涉及文件，确认对任务描述的理解没有歧义
- 如果有任何假设、不确定的接口行为或业务逻辑，**先在工作区写入假设清单（如 `assumptions.md`）并向我提问，不要直接实现**

### 1. 代码质量铁律
- **简洁至上**：禁止引入未明确要求的新依赖；禁止为单一用例创建抽象基类、Protocol、复杂 Enum 组合；能用 50 行解决的不用 500 行
- **手术式修改**：只修改与当前 task 直接相关的代码；不修无关的注释、格式、类型提示、引号风格；你的修改导致的废弃 import/变量必须删除，但历史遗留死代码不动
- **匹配现有风格**：优先拷贝-修改现有代码模式，而不是"重构得更通用"

### 2. 验证循环（Goal-Driven Execution）
- **如果是修复类任务**：先写一个能复现问题的测试，确认它失败，再修复，最后确认测试通过
- **如果是功能类任务**：先写测试或最小可运行脚本，再实现功能，最后确认通过
- **多文件任务**：每改 1-2 个文件后就要运行相关测试，禁止一次性改完所有文件再统一验证

### 3. 工程规范
- 在隔离工作区中实现上述任务
- 如有 typecheck / lint / test 配置，运行并保证通过
- 在 `.zeus/v3/ai-logs/` 目录写入 ai-log 文件（如 `T-V3-XXX.md`）
- **完成后必须**在当前工作区根目录写入 `zeus-result.json`，格式如下：
  ```json
  {
    "status": "completed",
    "changed_files": ["src/foo.py"],
    "test_summary": {"passed": 5, "failed": 0, "skipped": 0},
    "commit_sha": "abc1234",
    "artifacts": {}
  }
  ```
- **不需要**在隔离工作区中执行 `git commit`，也**不需要**修改主项目的 `task.json`

### 4. 进度上报
在执行过程中，每完成一个重要里程碑，请向当前工作区根目录的 `progress.jsonl` 追加一行 JSON：
```json
{"ts": "2026-04-13T15:30:00Z", "step": "planning", "message": "已完成架构分析"}
```
建议的 step 值：`planning`（规划）、`reading`（阅读）、`writing`（编写）、`testing`（测试）、`reviewing`（审查）、`completed`（完成）。

你也可以通过 HTTP POST 主动上报进度：
```bash
curl -X POST http://127.0.0.1:8000/tasks/T-V3-XXX/progress -H "Content-Type: application/json" -d '{"step":"writing","message":"正在修改 api/server.py"}'
```

完成后，请在 Kimi Code 主会话中报告执行结果。
```

## 🚀 开工流程

**写代码前必须按顺序执行：**

0. 回答上面的「动手前四问」（如有歧义/困惑，先向用户/调度器提问）
1. `pwd` — 确认当前在项目根目录
2. 读取 `claude-progress.md`（或 `.zeus/main/evolution.md`）— 了解最新状态
3. 读取 `.zeus/main/task.json` — 查看待完成任务
4. `git log --oneline -5` — 查看最近提交
5. 运行状态检查：`python .zeus/scripts/zeus_runner.py --status`
6. 如果基础验证失败，先修复基础状态

⚠️ **如果状态异常，先修复，不要继续！**

## 📋 工作规则

- **一次只做一个功能（一个 task）**
- 不要因为"代码已经写了"就把功能标记为完成
- 实现过程中不要悄悄改弱验证规则
- 优先依赖仓库里的持久化文件，而不是聊天记录

### 防过度延伸
- 如果发现"需要顺便做"的其他功能：
  1. 记录下来
  2. 添加到 `task.json` 或 `feature_list.json`
  3. **不要开始做**
  4. 继续完成当前 task

## ✂️ 代码简洁性三原则（Simplicity First）

**默认规则：用最小代码解决当前问题。不为未来做 speculation。**

### 原则 1：不超需求加功能
- 只实现明确要求的内容，以及为了该功能"能跑通"所必需的最小配套代码。
- 不要提前实现"以后可能用得上的"配置项、开关、通用接口。

### 原则 2：不为一行代码建抽象
- 如果一个抽象（类、接口、策略模式、工厂）只为当前这一个用例服务，不要建。
- 直接写函数。当同一个逻辑真的出现了第 3 次重复时，再考虑提取。

### 原则 3：不写不可能触发的错误处理
- 不要为"理论上可能但实际永远不会发生"的情况写复杂的防御代码。
- 如果无法判断某个异常是否真会发生，先实现最简单的路径，并在注释中标记：`# TODO: 评估此处是否需要处理 X 异常`。

### 自检问题
在提交任何代码前，问自己：
> "如果一位资深工程师 review 这段代码，他会不会说'这太过度设计了'？"
> 如果答案是"会"，立即简化。

## 🔪 手术式修改规范（Surgical Changes）

**核心原则：只碰必须碰的。只清理自己制造的。**

### 规则 1：禁止顺手改进相邻代码
当任务要求"修复 A"时：
- ❌ 不要重构与 A 无关的函数 B
- ❌ 不要为相邻的类添加类型提示
- ❌ 不要修改无关的注释和文档字符串
- ❌ 不要调整引号风格、缩进或换行

### 规则 2：匹配现有风格
即使你认为另一种风格更好，也要匹配当前文件的：
- 引号风格（`'` vs `"`）
- 类型提示使用习惯（有或无）
- 异常处理方式
- 命名惯例

### 规则 3：orphan 清理
- 如果你的修改导致某些 import / 变量 / 函数不再被使用 → **必须删除**
- 预先存在的死代码 / 未使用 import → **不要动**（除非 task 明确要求清理）

### 规则 4：diff 自检
在标记任务完成之前，必须在脑海中检查：
> "每一行改动是否都能直接追溯到用户的这个请求？"
> 如果有任何一行不能，撤销它。

## 🎯 验证循环工作法（Goal-Driven Execution）

**核心原则：把指令转化为可验证的目标，循环直到通过。**

### 指令转换表

| 收到的指令 | 必须转化成的目标 |
|-----------|-----------------|
| "修复 bug" | 1) 写复现测试 → 2) 确认失败 → 3) 修复 → 4) 确认通过 |
| "添加功能" | 1) 明确输入输出 → 2) 写测试 → 3) 实现 → 4) 测试通过 |
| "重构 X" | 1) 记录现有通过数 → 2) 重构 → 3) 确保通过数不变 |
| "优化性能" | 1) 测量基准 → 2) 优化 → 3) 再次测量确认提升 |

### 多步骤任务的计划格式

对于涉及多个文件的复杂任务，在动手前必须给出简要计划：

```
1. [步骤描述] → 验证：[如何确认这一步成功]
2. [步骤描述] → 验证：[如何确认这一步成功]
3. [步骤描述] → 验证：[如何确认这一步成功]
```

**禁止行为：**
- ❌ "我先看看代码，然后改改看"
- ❌ 一次性修改 5 个文件后再统一测试
- ❌ 用"应该可以了"代替实际验证

## ✅ 完成定义

一个 task 只有在以下条件**全部满足**时才算完成：

- [ ] 目标行为已经实现
- [ ] **修复类 task**：已先写复现测试，测试从"失败"变"通过"
- [ ] **功能类 task**：已写对应测试/可运行脚本，且通过
- [ ] 测试全部通过
- [ ] Lint / 类型检查无错误
- [ ] `task.json` 已更新（`passes: true`, `commit_sha` 已填）
- [ ] AI log 已写入 `.zeus/main/ai-logs/`
- [ ] 代码已 `git commit`（格式：`feat(T-001): description`）
- [ ] `zeus_runner.py --status` 显示正常
- [ ] **diff 自检通过**：每一行改动都能直接追溯到当前 task 需求

### 5.2 改进后的 PROMPT.md 模板（适用于所有 v3 子 Agent）

```markdown
# ZeusOpen v3 Agent Task Prompt

## 项目信息
- 项目名称：zeus-open
- 北极星指标：adoption_rate
- Zeus 版本：v3

## 当前任务
- 任务 ID：{{TASK_ID}}
- 所属 Wave：{{WAVE}}
- 类型：{{TYPE}}
- 标题：{{TITLE}}
- 描述：{{DESCRIPTION}}
- 涉及文件：{{FILES}}
- 依赖任务：{{DEPENDS_ON}}

## 执行要求

### 0. 认知校准（Think Before Coding）
在动手写代码前，你必须：
- 阅读涉及文件，确认对任务描述的理解没有歧义
- 如果有任何假设、不确定的接口行为或业务逻辑，**先在工作区写入假设清单（如 `assumptions.md`）并向我提问，不要直接实现**

### 1. 代码质量铁律
- **简洁至上**：禁止引入未明确要求的新依赖；禁止为单一用例创建抽象基类、Protocol、复杂 Enum 组合；能用 50 行解决的不用 500 行
- **手术式修改**：只修改与当前 task 直接相关的代码；不修无关的注释、格式、类型提示、引号风格；你的修改导致的废弃 import/变量必须删除，但历史遗留死代码不动
- **匹配现有风格**：优先拷贝-修改现有代码模式，而不是"重构得更通用"

### 2. 验证循环（Goal-Driven Execution）
- **如果是修复类任务**：先写一个能复现问题的测试，确认它失败，再修复，最后确认测试通过
- **如果是功能类任务**：先写测试或最小可运行脚本，再实现功能，最后确认通过
- **多文件任务**：每改 1-2 个文件后就要运行相关测试，禁止一次性改完所有文件再统一验证
- 多步骤任务需在 ai-log 中按 `1. [步骤] → 验证：[检查点]` 格式记录计划

### 3. 工程规范
- 在隔离工作区中实现上述任务
- 如有 typecheck / lint / test 配置，运行并保证通过
- 在 `.zeus/v3/ai-logs/` 目录写入 ai-log 文件（如 `{{TASK_ID}}.md`）
- **完成后必须**在当前工作区根目录写入 `zeus-result.json`，格式如下：
  ```json
  {
    "status": "completed",
    "changed_files": ["src/foo.py"],
    "test_summary": {"passed": 5, "failed": 0, "skipped": 0},
    "commit_sha": "abc1234",
    "artifacts": {}
  }
  ```
- **不需要**在隔离工作区中执行 `git commit`，也**不需要**修改主项目的 `task.json`

### 4. 进度上报
在执行过程中，每完成一个重要里程碑，请向当前工作区根目录的 `progress.jsonl` 追加一行 JSON：
```json
{"ts": "2026-04-13T15:30:00Z", "step": "planning", "message": "已完成架构分析"}
```
建议的 step 值：`planning`（规划）、`reading`（阅读）、`writing`（编写）、`testing`（测试）、`reviewing`（审查）、`completed`（完成）。

你也可以通过 HTTP POST 主动上报进度：
```bash
curl -X POST http://127.0.0.1:8000/tasks/{{TASK_ID}}/progress -H "Content-Type: application/json" -d '{"step":"writing","message":"正在修改..."}'
```

完成后，请在 Kimi Code 主会话中报告执行结果。
```

## 六、实施路线图

建议按以下优先级逐步应用改进，避免一次性变更过大：

### 阶段 1：立即生效（Think Before Coding）
- 在 AGENTS.md 开头插入「动手前四问」
- 在 PROMPT.md 的「执行要求」中增加「0. 认知校准」
- **预期效果**：减少因理解偏差导致的返工

### 阶段 2：一周后生效（Surgical Changes）
- 在 AGENTS.md 中新增「手术式修改规范」
- 在 PROMPT.md 中增加对应的「手术式修改」铁律
- **预期效果**：diff 更干净，多 Agent 并行冲突减少

### 阶段 3：两周后生效（Goal-Driven Execution）
- 在 AGENTS.md 中新增「验证循环工作法」
- 在 PROMPT.md 中增加「验证循环」要求
- 修改完成定义，加入"测试先行"和"diff 自检"
- **预期效果**：代码质量提升，bug 遗漏减少

### 阶段 4：一个月后生效（Simplicity First）
- 在 AGENTS.md 中新增「代码简洁性三原则」
- 在 PROMPT.md 中明确禁止为单一用例建抽象
- **预期效果**：代码库膨胀速度放缓，维护成本降低

## 七、效果评估指标

在实施改进 2-4 周后，可通过以下数据评估效果：

| 指标 | 如何测量 | 预期变化 |
|------|---------|---------|
| **任务返工率** | 同一个 task 被重新打开修改的次数 / 总 task 数 | 下降 |
| **diff 干净度** | zeus-result.json 中 `changed_files` 里出现无关文件的比例 | 下降 |
| **测试先行率** | bug 修复/功能开发中，ai-log 提到"先写测试"的比例 | 上升 |
| **平均代码量** | 每个 task 新增/修改的代码行数中位数 | 下降或持平 |
| ** blocker 率** | 因"理解偏差"或"改动范围过大"导致标记为 false 的比例 | 下降 |

## 八、结论

ZeusOpen v3 的 Agent 设定在**流程管理、任务追踪、可观测性**方面已经相当完善，但在**代码质量与认知行为**这一核心维度上存在明显盲区。通过将 Andrej Karpathy 的四大原则系统地嵌入到 `AGENTS.md` 和 `PROMPT.md` 中，可以：

1. **降低返工率**：让子 Agent 在动手前就澄清假设和歧义
2. **降低合并冲突**：通过手术式修改规范约束 diff 范围
3. **提升代码质量**：通过简洁性原则遏制 LLM 的过度工程本能
4. **增强可验证性**：通过目标驱动执行培养测试先行的文化

这不仅是对文档的升级，更是对 ZeusOpen v3 多 Agent 协作体系可靠性的根本性增强。
