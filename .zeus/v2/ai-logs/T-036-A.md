# T-036-A AI Log

## Decision Rationale

- **SQLite over JSON/JSONL**: 调度器状态需要在服务器重启后快速恢复，且包含结构化查询需求（如按 agent_id 过滤活跃任务、按 read 状态过滤邮件）。SQLite 原生支持 ACID 事务和并发读，比纯文本 JSONL 更适合作为持久化核心。
- **三张表设计（meta / active_tasks / mailbox）**:
  - `meta` 存储全局标量/配置（如 current_wave、scheduling_mode），值统一用 JSON 文本保存，保留复杂对象（dict/list/bool/null）的往返能力。
  - `active_tasks` 以 `task_id` 为主键，记录运行中任务的核心字段，便于重启后直接恢复调度上下文。
  - `mailbox` 使用自增 `id` 主键，支持消息的去重、已读标记和按收件人查询，为后续 Agent Mailbox 协议提供底层存储。
- **save/load API 采用全量替换（DELETE + INSERT）**：调度器状态快照通常一起产生、一起消费，全量替换实现简单、原子性强，且当前数据规模（单 wave 任务数 < 100）下性能足够。未来若需要增量更新，可在 `SchedulerStateDB` 上扩展 `upsert_task` / `append_mail` 等方法。
- **复用已有的 `scheduler_state.py` 骨架**：工作区中已存在实现完整的 `SchedulerStateDB` 类，包含 `_ensure_tables`、`save`、`load`、`clear`、`set_meta`、`get_meta` 等 API。经代码审查确认其设计符合 KISS 原则，无需重写，只需补齐单元测试。

## Target Impact

- **北极星指标**: `developer_adoption_rate`
- **贡献方式**: SQLite 持久化核心消除了服务器重启后调度状态丢失的风险，使 Zeus v2  backend 具备生产级可靠性。对开发者而言，这意味着：
  1. 可以安全地重启服务而无需手动恢复 wave/task 状态；
  2. 多 Agent 协作的 mailbox 消息在重启后仍然保留，降低调试和运维成本。
- **量化估算**:

  - 间接提升开发者对 Zeus 框架作为"可长期运行工具"的信任度，从而促进采纳率。
