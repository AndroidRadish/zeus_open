# T-011 — Extend task.json schema and store.py with adaptive wave fields

## Decision Rationale

To implement adaptive wave rescheduling, the orchestrator needs to distinguish between:
- the wave a task was originally planned in (`original_wave`)
- the wave it is currently scheduled to run in (`scheduled_wave`)
- the previous wave it was moved from, for audit (`rescheduled_from`)

Storing these fields directly in `task.json` keeps the single source of truth intact and avoids introducing a secondary scheduling database. The fields are optional with `null` defaults, preserving backward compatibility for v1/main consumers.

For atomic batch updates to `task.json` (required when the orchestrator reschedules multiple tasks under a single file lock), `store.py` needed a new `update_json_fields` helper. We chose a list-oriented API (`list_key`, `id_field`, `updates`) because `task.json` is the primary consumer and it stores tasks in a top-level list.

## Execution Summary

- Updated `.zeus/schemas/task.schema.json`:
  - Added `original_wave`, `scheduled_wave`, `rescheduled_from` as optional integer/null fields.
  - Clarified that `wave` now acts as the runtime alias of `scheduled_wave`.
- Updated `.zeus/v2/scripts/store.py`:
  - Added `update_json_fields` to `AbstractStore` and all concrete implementations (`LocalStore`, `TencentCosStore`, `RedisStore`).
  - `LocalStore` implementation uses the existing `filelock`-based `lock()` to ensure atomic read-modify-write cycles.
- Updated `.zeus/v2/scripts/test_store.py`:
  - Added `test_local_store_update_json_fields` for functional coverage.
  - Added `test_local_store_update_json_fields_concurrent` for thread-safety validation (5 threads × 4 updates each).
- All 5 store tests pass.

## Target Impact

This is a pure infrastructure task; no direct user-facing impact yet. It unlocks `multi_agent_efficiency` gains in T-012 (orchestrator scheduling loop refactor) by providing the data model and storage primitives required for dynamic wave rescheduling.
