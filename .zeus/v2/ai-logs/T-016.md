# T-016 — Refactor zeus_orchestrator.py to delegate execution to dispatcher and add config parsing

## Decision Rationale

The orchestrator was previously responsible for both workspace preparation and task completion signaling. With the new dispatcher layer, the orchestrator should focus on **scheduling and workspace setup**, while the dispatcher owns **execution and lifecycle events**.

This separation makes it possible to swap execution backends (mock → Kimi → Claude) without touching the core wave-scheduling logic.

## Execution Summary

- Refactored `zeus_orchestrator.py::dispatch_task`:
  - Added `_build_dispatcher(store)` which reads `config.subagent.dispatcher` and `config.subagent.timeout_seconds` and delegates to `build_dispatcher()`.
  - Workspace preparation (clean, copy, write PROMPT.md) remains in `dispatch_task`.
  - Actual execution is handed off to `dispatcher.run(task, workspace_path, prompt, bus)`.
  - Removed the old immediate `task.completed` emission from `dispatch_task`; the dispatcher now emits `started` / `completed` / `failed` events.
- Updated `await_wave_completion` to recognize dict outcomes with `status == "failed"` (in addition to raw exceptions) so CLI non-zero exits are counted as wave failures.
- Fixed all test fixtures (`test_orchestrator.py`, `test_integration.py`) to set `"subagent": {"dispatcher": "mock"}` in `config.json`, preventing accidental real `kimi` invocations during test runs (the host machine has Kimi CLI installed).

## Target Impact

- `multi_agent_efficiency` ↑↑: the orchestrator can now truly dispatch parallel subagents instead of just writing prompt files.
- `developer_adoption_rate` ↑: Kimi/Claude users only need to set one config key (or leave it on auto) to enable full automation.
