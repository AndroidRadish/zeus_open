# T-035 — 任务生命周期控制（retry / cancel / pause）API + Web UI

## Decision Rationale

1. **状态模型扩展**：在 `task.schema.json` 中引入 `status` 字段，枚举值为 `pending|running|paused|completed|failed|cancelled`，与 `passes` 布尔字段互补，提供更细粒度的生命周期表达。
2. **原子更新**：新增 `_update_task_status()` 辅助函数，在 `store.lock()` 保护下原子读写 `task.json`，避免并发调度导致的状态竞争。
3. **调度器感知**：在 `_get_ready_tasks()` 和 `_get_global_ready_tasks()` 中过滤 `paused|cancelled|completed` 状态的任务，确保被 pause/cancel 的任务不会被继续调度。
4. **状态联动**：`_run_and_capture()` 在任务开始/结束/异常时自动更新 `status`，保证调度器状态和实际执行结果一致。
5. **API 端点**：在 `zeus_server.py` 新增 `POST /tasks/{task_id}/retry|cancel|pause`，由 `ZeusOrchestrator` 的生命周期方法直接处理，保持后端接口薄、业务逻辑下沉在调度器层。
6. **Web UI**：Global Execution 页面为运行中任务添加 Pause/Cancel 按钮，pending 任务添加 Cancel 按钮，quarantine 任务添加 Retry 按钮，并引入 `status_map` 实时展示状态 badge。
7. **复制安全**：`dispatch_task` 中的 `shutil.copytree` 增加 `*.tmp` 和 `*.lock` 忽略规则，避免 `_update_task_status` 写入的临时文件/锁文件在并发复制时触发 `WinError 2` 竞争。

## Target Impact

- **developer_adoption_rate**：任务生命周期控制是生产级调度器的基础能力，降低了用户因单任务失败而重启整个 wave 的门槛，提升框架可维护性和采用意愿。
