# T-036 — Production Hardening: Graceful Shutdown & State Recovery

## Decision Rationale

T-036 was split into five sub-tasks (A/B/C/D/E) to allow parallel work on SQLite persistence, signal handling, orchestrator resume logic, Web UI recovery banners, and integration tests. Sub-agents T-036-B and T-036-C launched via the global scheduler crashed silently due to dispatcher/encoding bugs, leaving their workspaces empty and their `task.json` statuses stuck in `running`. Rather than waiting indefinitely for phantom agents, the parent session took direct ownership and implemented the missing B/C logic in a single pass, then completed E.

Key design choices:
- **SQLite as the sole source of truth for scheduler survival**: `scheduler_state.db` stores `meta.scheduler_active` plus the `active_tasks` list. It is cheap, file-based, and requires no extra services.
- **Graceful shutdown via event loop + signal handlers**: `ZeusOrchestrator.stop_global_scheduler()` sets an internal `_shutdown` flag and persists the current running task set. `zeus_server.py` wires this into both FastAPI `shutdown` events and `SIGINT`/`SIGTERM` signal handlers.
- **Resume on startup without auto-restarting**: The server `startup` event detects `scheduler_active`, calls `resume_from_state()` to rewrite `task.json` statuses back to `running`, and lets the UI reflect the recovered state. We deliberately do not auto-launch the global scheduler on startup to avoid surprising side effects.
- **Clear on normal completion**: `run_global()` clears the SQLite snapshot in its `finally` block so a clean exit does not trigger a false recovery on the next server start.
