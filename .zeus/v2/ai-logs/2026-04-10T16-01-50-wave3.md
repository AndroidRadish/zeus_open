## Decision Rationale

### PyQt vs WebEngine
The design spec originally suggested `QWebEngineView` for the Graph Viewer, but `PyQt6.QtWebEngineWidgets` is not installed in the current Windows environment. To avoid forcing an extra dependency (`PyQt6-WebEngine`) on users, I switched to `PyQt6.QtSvgWidgets.QSvgWidget`, which renders the SVG returned by `/graph/svg` natively. This keeps the GUI dependency footprint minimal (only PyQt6 core + QtSvgWidgets).

### API Client
I used `requests` for HTTP because it is already installed in the environment and widely available. The client is thin and wraps the five endpoints we need for the GUI: status, wave, agents, discussion, and graph/svg.

### Approval Gate
The GUI polls every 2 seconds. When all tasks in the current wave are `passes: true`, a modal `ApprovalDialog` pops up with three choices:
1. **Approve next wave** — calls `POST /wave/{n}/approve` and advances `meta.current_wave`.
2. **Pause** — closes the dialog and leaves the user in control.
3. **Retry failed tasks** — informs the user to use the CLI orchestrator for retries.

A guard (`_approval_shown_for_wave`) prevents the dialog from spamming the user on subsequent polls.

### Web Frontend (T-007)
`index.html` was already present in the repository (committed during Wave 2). I reviewed it against the acceptance criteria:
- Single file, zero build step ✅
- Vue 3 + Tailwind CSS + Mermaid.js via CDN ✅
- Polling every 2s to `/status`, `/wave/{n}`, `/agents` ✅
- Read-only dashboard with status, tasks, agents, discussion, and graph tabs ✅
- No remote approval controls by default ✅

Therefore T-007 is considered complete and only needed task-state bookkeeping.
