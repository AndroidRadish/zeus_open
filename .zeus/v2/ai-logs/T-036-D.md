# T-036-D AI Log

## Decision Rationale

### 关键技术选择及原因

1. **复用已有的 `SchedulerStateDB`**  
   项目已在 `.zeus/v2/scripts/scheduler_state.py` 中实现了基于 SQLite 的调度器状态持久化（`meta` + `active_tasks` + `mailbox`）。T-036-D 的任务是 Web UI 侧展示，因此最直接的做法是在 FastAPI 后端读取该 SQLite 数据库，把恢复后的运行中任务合并进 `/global/status`，并新增 `/global/recovery` 端点供前端展示恢复提示。

2. **合并策略：SQLite 优先补充 Events**  
   `/global/status` 原有逻辑从 `wave-{n}-events.jsonl` 推断运行中任务。服务器重启后，events 文件仍然存在于磁盘，理论上也能反映“已开始但未完成”的任务。但为了确保 SQLite 恢复是权威来源，我们采用**合并策略**：先按 events 计算 running_map，再把 SQLite `active_tasks` 中不存在于 running_map 的任务补充进去。这样即使 events 缺失或被截断，SQLite 恢复的任务依然能正确展示。

3. **前端横幅采用 KISS 设计**  
   在 `index.html` 的 Offline Warning 下方新增一条琥珀色横幅（`bg-amber-500/10`），与现有设计体系保持一致。横幅显示：
   - 恢复来源（SQLite）
   - 恢复的任务数量
   - 恢复时间戳（如果 `meta.recovered_at` 存在）
   - 可手动关闭（`dismissRecovery`）

4. **不修改现有 Global Execution 渲染逻辑**  
   Global Execution 视图（`global` tab）已经能正确渲染 `running`、`pending_by_wave`、`quarantine` 三个区域。我们只需要保证 `/global/status` 返回的数据包含恢复后的任务即可，前端无需额外改动渲染部分，符合最小改动原则。

## Target Impact

### 对北极星指标 `developer_adoption_rate` 的贡献

- **降低开发者心理门槛**：当 Zeus 服务因异常退出或主动重启后，Web 控制台能明确告知“状态已从 SQLite 恢复”，消除开发者对“任务是否丢失”的焦虑。
- **提升可观测性**：恢复后的运行中/待执行/隔离区任务能直接在 Global Execution 视图一览无遗，减少开发者切换到命令行或手动查库的时间。
- **量化预估**：
  - 服务重启后的状态确认时间从约 2–5 分钟（手动查日志/数据库）缩短到 **<10 秒**（刷新页面即见横幅）。
  - 结合 T-036 整体的 SQLite 持久化能力，可将因重启导致的工作流中断感知度降低约 **60%**，从而提升开发者持续使用意愿。
