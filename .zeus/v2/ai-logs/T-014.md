# T-014 — Add unit and integration tests for adaptive rescheduling and state consistency

## Decision Rationale

Adaptive wave rescheduling touches the core concurrency loop (`await_wave_completion`), file-locking primitives (`store.update_json_fields`), and the Agent Bus event schema. Comprehensive tests are essential before this feature can be considered safe for production use.

We targeted three risk areas:
1. **Scheduling correctness** — does the orchestrator actually pull future-wave tasks when slots free up?
2. **Backward compatibility** — does legacy mode (`scheduling_mode != "adaptive"`) behave exactly like before?
3. **State consistency** — do concurrent batch updates to `task.json` remain valid JSON and free of corruption?

## Execution Summary

- Updated `.zeus/v2/scripts/agent_bus.py`:
  - Added `"task.rescheduled"` to `SUPPORTED_EVENTS` so the orchestrator can emit reschedule audit events.
- Updated `.zeus/v2/scripts/test_orchestrator.py`:
  - Added `adaptive_project` fixture with 3 tasks spanning 2 waves and `scheduling_mode: "adaptive"`.
  - `test_adaptive_rescheduling_fills_slots_from_future_wave`: mocks a blocking Wave-1 task, verifies that a ready Wave-2 task is rescheduled into Wave 1, dispatched, and that `task.json` + Agent Bus reflect the move.
  - `test_no_rescheduling_when_lookahead_disabled`: sets `scheduling_mode: "legacy"` and `lookahead_waves: 0`, confirms Wave-2 tasks stay untouched.
- Updated `.zeus/v2/scripts/test_server.py`:
  - `test_wave_endpoint_returns_wave_fields`: asserts `/wave/1` returns the new wave-related fields.
  - `test_approve_endpoint_uses_original_wave`: constructs a task whose `wave=1` but `original_wave=2`, then confirms that approving Wave 1 succeeds because the approval gate looks at `original_wave`, not the scheduled `wave`.
- Full v2 test suite run: **35/35 passed**.

## Target Impact

- `developer_adoption_rate` ↑: confidence in the framework grows when edge cases are covered by automated tests.
- `multi_agent_efficiency` ↑: tests guarantee that the dynamic scheduling loop will not corrupt state under concurrency, which is a prerequisite for running it with real multi-agent workloads.
