# T-030 — Implement GlobalScheduler with quarantine and wave-as-view semantics

## Decision Rationale

The existing `await_wave_completion` is strictly wave-locked. Even with adaptive slot filling (M-005), execution cannot cross wave boundaries. For long-running projects with many waves, this creates an artificial throughput ceiling.

**Key design choices:**
- **Preserve backward compatibility**: `await_wave_completion` remains untouched. A new `run_global()` method provides the global scheduling semantics.
- **Wave as a view, not a gate**: `wave`, `original_wave`, and `scheduled_wave` are kept purely for UI grouping, reporting, and phase-aware navigation. They do not influence dispatch readiness.
- **Quarantine over immediate failure propagation**: When a task fails, it is moved to a `quarantine` array in `task.json`. Tasks that do **not** transitively depend on the quarantined task can still proceed. This prevents a single failing agent from stalling the entire project.
- **Dependency-driven readiness**: `_get_global_ready_tasks` uses the full dependency graph. A task is ready only when all its direct dependencies have passed and none are quarantined.

## Target Impact

- **multi_agent_efficiency** ↑↑↑: Tasks from future waves can execute as soon as their dependencies are satisfied, maximizing concurrency slot utilization.
- **developer_adoption_rate** ↑↑: Large projects no longer suffer from wave-by-wave idle time. The quarantine zone provides clear visibility into failures without halting unrelated progress.
- **observability** ↑↑: The `quarantine` array is queryable by the Web UI and gives a persistent, auditable record of every failed task together with its workspace path and failure reason.
